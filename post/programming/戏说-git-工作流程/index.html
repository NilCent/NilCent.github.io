戏说 Git 工作流程

Feb 01, 2023
1 minute read

Git 被称为版本管理系统, 简单来说就是可以存储项目在任意时刻的快照. 在使用 Git 时, 一个十分重要的问题就是该怎么组织这些快照. 也就是这篇文章要讨论的话题 – Git 工作流程, 一个不属于 Git 的 Git 技巧.

在这篇文章, 我将通过个人博客的例子来讲一下我的 Git 工作流程.

需求

个人博客的项目结构如下:

1
2
3

/myblog
    /posts  
    deploy.sh
  • posts 目录下存储写的文章
  • deploy.sh 是将文章发布到平台的脚本

对个人博客我有以下几个需求:

  • 写完的文章可以自动发布(CI)
  • 每月要写完四篇文章(Feature)
  • 错别字之类的问题要立刻修复(Hotfix)

可以看到, 绝大多数项目都是在解决和这个博客相似的问题.

master 分支

在我的这个项目中, 文章会有完成和草稿两种状态. 我只希望发布完成的文章, 所以我需要一个分支来专门存储和发布完成的文章, 也就是 master 分支.

master 分支负责管理生产环境的代码, 只会把开发且测试完的代码合到 master 分支上. 通常还会将项目的编译部署之类的工作交给 master 来做, 每当合并或推送代码时, 自动执行指定的脚本.

topic 分支

我在写作时, 通常会同时写多篇文章. 比如说, 我会在月初选好四个题目, 每一篇都写点内容. 为了避免代码冲突, 也是图一个方便, 很多时候我更倾向与将这些文章放在同一个分支内. 但是很有可能会出现像是写完的文章不能立刻发布, 写不完的文章只能删掉之类的问题.

一个比较符合逻辑的做法是 – 一篇文章(Feature)一个分支, 也就是 topic 分支.

在实际项目中, 功能提前上线或功能取消是很常见的事情. 绝不能将不同的功能合到同一个分支中开发.

dev 分支

在使用 topic 分支后, 一个显著的问题就是合并会变麻烦, 我不希望在 master 分支上解决合并冲突, 这就一个需要专门负责开发的分支 – dev 分支.

topic 分支只由 dev 分支派生, 完成功能后, 合并回 dev 分支.

master 分支要尽可能以 fast-forward 的方式合并 dev 分支.

除此之外, 测试工作一般也在 dev 分支上进行. 当在开发阶段发现 bug 时, 把它当作一个新的 topic 处理, 不要在 dev 上直接改动.

hotfix 分支

生产环境的大多数错误, 是不能等到下一个版本再解决的. 这时就需要从 master 分支派生出用于修复 bug 的分支, 也就是 hotfix 分支.

要注意的是, 测试完 hotfix 分支后, 要合并到 master 和 dev 两个分支上.

综上

简单来说, 我将上述的四种分支分为了两类:

  • 主分支(伴随整个项目开发周期, 只合并和派生分支):
    • master 负责发布生产环境代码
    • dev 负责开发和测试工作
  • 开发分支(写代码的分支, 被合并后生命周期结束):
    • topic 实现功能的场所
    • hotfix 解决生产环境 bug 的场所

这也就是我的 Git 工作流程.
